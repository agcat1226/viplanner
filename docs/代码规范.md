# VIPlanner 代码与 PyTorch 工程规范

本规范用于统一 VIPlanner 项目的编码风格、PyTorch 工程实践与协作要求，重点防止多模态、多线程深度学习项目中常见的三类问题：**内存泄漏（OOM）**、**维度灾难（Shape Mismatch）** 与 **静默失败（Silent Failure）**。

适用范围：
- 所有 Python / PyTorch 训练、推理、评测与部署代码
- 所有多模态输入处理、扩散采样、SDF 引导、并发建图与规划模块
- 所有提交到主分支或实验分支、最终会被团队复用的代码

---

## 1. 设计原则

### 1.1 可读性优先于技巧
深度学习项目中的大多数问题并不是“不会写”，而是“几周后没人敢改”。因此：
- 优先选择**语义明确**的命名，而不是短小但模糊的缩写
- 优先写出**显式维度变换**，而不是依赖隐式广播或难以理解的 `reshape`
- 优先用清晰的中间变量表达计算过程，而不是一行链式写完

### 1.2 显式优先于隐式
任何会影响训练稳定性、显存占用、线程安全、设备迁移的行为，都必须写清楚。
- 显式声明张量形状
- 显式切换 `grad/no_grad`
- 显式控制设备与 dtype
- 显式记录配置来源
- 显式处理异常值与边界条件

### 1.3 默认防御式编程
对外部输入、模型输出、梯度、配置参数、线程共享状态，一律按“不可靠”处理。
- 对张量 shape 做断言
- 对关键数值做有限值校验
- 对共享状态加锁或复制快照
- 对实验性逻辑提供 fallback

## 2. 张量维度与类型注释规范（Strict Tensor Documentation）

深度学习代码中最难维护的部分通常不是算法本身，而是**张量维度在模块间流动时的含义丢失**。本项目强制要求：

1. 所有核心模块的 `forward`、损失函数、采样函数、数据预处理函数，必须在 Docstring 中标注输入输出 shape
2. 复杂维度变换必须在代码旁写注释
3. 变量名要体现语义，不允许大面积使用 `x`, `y`, `tmp`, `feat2` 之类的模糊命名
4. 对 batch/time/channel/trajectory/token 等维度，命名要统一

### 2.1 推荐维度命名约定

| 符号 | 含义 |
|---|---|
| `B` | batch size |
| `K` | temporal window / history length |
| `C` | channel |
| `H, W` | image height / width |
| `N` | 点数 / token 数 / 通用序列长度 |
| `T` | 轨迹长度 / diffusion steps 中的轨迹维 |
| `D` | 特征维度 |
| `D_model` | 主干隐藏维度 |
| `M` | 候选轨迹数 / 模态数 |

### 2.2 Docstring 强制格式

**✅ 推荐做法：**

```python
import torch
import torch.nn as nn

class TemporalEncoder(nn.Module):
    def forward(self, rgb_seq: torch.Tensor, depth_seq: torch.Tensor) -> torch.Tensor:
        """
        提取时序滑动窗口的视觉上下文特征。

        Args:
            rgb_seq: 时序 RGB 图像张量。Shape: (B, K, C, H, W)
            depth_seq: 时序深度图像张量。Shape: (B, K, 1, H, W)

        Returns:
            context: 融合时序信息的上下文向量。Shape: (B, D_model)
        """
        b, k, c, h, w = rgb_seq.shape
        # (B, K, C, H, W) -> (B*K, C, H, W)
        rgb_flat = rgb_seq.view(b * k, c, h, w)
        return rgb_flat.mean(dim=(2, 3)).view(b, k, c).mean(dim=1)
```

### 2.3 复杂变换必须写局部注释

以下情况必须加注释：
- `view / reshape / permute / transpose / flatten / squeeze / unsqueeze`
- 拼接、分块、交错采样
- 从 `(B, T, D)` 展平到 `(B*T, D)`
- 从 patch/token 恢复空间结构

**✅ 推荐写法：**

```python
# (B, T, D) -> (B*T, D)
traj_flat = traj_emb.reshape(batch_size * horizon, hidden_dim)

# (B, N, H, W) -> (B, N, H*W)
feature_map = feature_map.flatten(start_dim=2)

# (B, K, C, H, W) -> (B, C, K, H, W)
rgb_seq = rgb_seq.permute(0, 2, 1, 3, 4)
```

### 2.4 禁止模糊 reshape

**❌ 严禁：**

```python
def forward(self, x):
    x = x.reshape(-1, 256)
    return self.layer(x)
```

问题：
- `-1` 掩盖真实语义
- 无法判断 batch 是否被打散
- 未来一旦输入 shape 改变，极易静默出错

### 2.5 Shape 断言要求

在跨模块边界、训练入口、损失函数入口、采样逻辑入口，建议加入 shape 断言。

```python
def fuse_context(rgb_feat: torch.Tensor, depth_feat: torch.Tensor) -> torch.Tensor:
    assert rgb_feat.ndim == 3, f"rgb_feat ndim mismatch: {rgb_feat.shape}"
    assert depth_feat.ndim == 3, f"depth_feat ndim mismatch: {depth_feat.shape}"
    assert rgb_feat.shape[:2] == depth_feat.shape[:2], (
        f"batch/token mismatch: rgb={rgb_feat.shape}, depth={depth_feat.shape}"
    )
    return torch.cat([rgb_feat, depth_feat], dim=-1)
```

### 2.6 类型注解要求

- 所有公开函数必须写类型注解
- `torch.Tensor` 必须标注
- 可选参数使用 `Optional[...]`
- 配置对象使用明确类型而不是裸 `dict`

**✅ 推荐：**

```python
from typing import Optional


def compute_guidance(
    x_prev: torch.Tensor,
    sdf_value: torch.Tensor,
    safety_margin: float,
    mask: Optional[torch.Tensor] = None,
) -> torch.Tensor:
    ...
```

## 3. 推理期梯度干预安全规范（Inference Autograd Guardrails）

VIPlanner 的关键风险点在于：**扩散采样整体是推理逻辑，但其中局部步骤又需要对中间状态求梯度**。这个场景极易因为计算图残留造成 OOM。

### 3.1 顶层默认 `no_grad`

所有推理主流程，默认必须在 `@torch.no_grad()` 或 `with torch.no_grad():` 下运行。仅在真正需要局部梯度的片段中使用 `torch.enable_grad()`。

**✅ 推荐：**

```python
@torch.no_grad()
def guided_denoise_step(model, x_t, isdf_model, t, context, guidance_scale):
    predicted_noise = model(x_t, t, context)
    x_prev = remove_noise_formula(x_t, predicted_noise, t)

    with torch.enable_grad():
        x_prev = x_prev.detach().requires_grad_(True)
        sdf_val = isdf_model(x_prev)
        loss = compute_collision_penalty(sdf_val)
        gradient = torch.autograd.grad(loss, x_prev, retain_graph=False, create_graph=False)[0]

    x_guided = x_prev - guidance_scale * gradient.detach()
    return x_guided.detach()
```

### 3.2 局部求导后必须立刻断图

强制要求：
- `torch.autograd.grad(..., retain_graph=False)`
- 梯度使用前后都要考虑 `detach()`
- 返回下一个 diffusion step 的张量时，不能携带上一时刻图

**必须避免的错误：**
- 在 diffusion 循环中把带图的 `x_prev` 直接传给下一步
- 对同一张量连续 `backward()` 但不清理历史图
- 在调试中临时加了 `retain_graph=True` 后忘记删

### 3.3 不允许在推理中误保留训练图

推理阶段严禁：
- 调用会触发优化器状态更新的逻辑
- 使用 `.backward()` 替代 `torch.autograd.grad()` 求局部梯度
- 不经检查地混用 AMP、checkpoint 与局部求导

**推荐原则：**
- 推理期局部引导优先使用 `torch.autograd.grad`
- 除非确有必要，不要在引导步骤创建二阶图
- 若需要二阶信息，必须在 PR 说明显存成本与收益

### 3.4 梯度有限值校验机制

iSDF 在增量建图初期可能不稳定，产生 NaN/Inf 梯度。应用前必须检查。

```python
if not torch.isfinite(gradient).all():
    logger.warning("检测到无效 SDF 梯度（NaN/Inf），当前步跳过物理引导。")
    gradient = torch.zeros_like(gradient)
```

### 3.5 建议加入梯度范数裁剪或钳制

当梯度虽有限但极端大时，仍可能导致轨迹瞬间崩坏。建议：

```python
grad_norm = gradient.norm(dim=-1, keepdim=True).clamp_min(1e-6)
max_norm = 10.0
gradient = gradient * (max_norm / grad_norm).clamp(max=1.0)
```

### 3.6 显存敏感代码必须自查以下事项

提交前至少检查：
- 是否误用了 `retain_graph=True`
- 是否存在循环内部累积 Python list 保存带图 tensor
- 是否将日志、调试缓存保存成未 `.detach()` 的 tensor
- 是否在 `loss_dict` 中返回了带图对象供外部长期持有
- 是否在多个时间步重复构图但没有释放

## 4. NaN / Inf / Silent Failure 防御规范

静默失败比显式崩溃更危险，因为它会污染实验结论。

### 4.1 对关键中间量做有限值检查

以下对象建议强制检查：
- loss
- 梯度
- SDF 输出
- 轨迹坐标
- 扩散噪声预测
- 归一化前的方差、分母项

```python
def assert_finite(name: str, tensor: torch.Tensor) -> None:
    if not torch.isfinite(tensor).all():
        raise ValueError(f"{name} contains NaN/Inf: shape={tuple(tensor.shape)}")
```

### 4.2 禁止吞异常

**❌ 严禁：**

```python
try:
    output = model(x)
except Exception:
    return None
```

这会掩盖训练或推理问题。必须：
- 记录上下文
- 保留原始异常信息
- 仅在明确存在 fallback 机制时做降级

**✅ 推荐：**

```python
try:
    output = model(x)
except RuntimeError as e:
    logger.exception("Model forward failed. input_shape=%s", tuple(x.shape))
    raise
```

### 4.3 日志中必须带上下文

出现异常或警告时，日志至少包含：
- 模块名
- 当前 step / batch / time index
- 关键 tensor shape
- 关键阈值或配置值

## 5. 异步并发与权重共享规范（Concurrency Safety）

部署场景下，iSDF 建图线程会更新共享模型，而扩散规划线程会读取模型计算梯度。这类“边写边读”的模式是事故高发区。

### 5.1 禁止跨线程直接读取正在更新的权重

严禁出现以下模式：
- 一个线程在 `.backward()` / `optimizer.step()`
- 另一个线程同时对同一个模型对象做前向或求梯度

这可能导致：
- 读到半更新状态
- autograd 图污染
- 极难复现的偶发崩溃

### 5.2 必须使用双缓冲或锁

**✅ 推荐：锁 + 推理副本**

```python
import threading

isdf_lock = threading.Lock()

# 共享模型：仅建图线程写
isdf_shared_model = ...
# 推理模型：仅规划线程读
isdf_inference_model = ...


def mapping_worker():
    while True:
        loss = compute_mapping_loss(...)
        loss.backward()
        with isdf_lock:
            optimizer.step()
            optimizer.zero_grad(set_to_none=True)


def planning_worker():
    with isdf_lock:
        state_dict = {
            k: v.detach().cpu() if v.is_cuda else v.detach().clone()
            for k, v in isdf_shared_model.state_dict().items()
        }
    isdf_inference_model.load_state_dict(state_dict)
    # 后续只使用 isdf_inference_model 推理
```

### 5.3 共享状态必须定义读写边界

以下状态必须明确：
- 谁负责写
- 谁负责读
- 何时同步
- 同步粒度是全量参数、局部缓存还是只读快照

建议在模块头部注释说明线程模型。

### 5.4 不允许共享可变缓存对象

例如：
- 全局 list 保存最近轨迹
- 全局 dict 缓存中间特征
- 共享 tensor 原地修改

如确需共享，必须：
- 加锁
- 或复制快照
- 或改成消息队列传递不可变数据

## 6. 设备无关性编程（Device Agnosticism）

为了保证代码可以运行在 CPU、单卡、多卡、不同编号 GPU 或未来的加速设备上，禁止写死设备名。

### 6.1 禁止硬编码 `.cuda()`

**❌ 严禁：**

```python
noise = torch.randn(x_t.shape).cuda()
```

问题：
- 当模型在 CPU 上运行时直接崩溃
- 当主设备是 `cuda:1` 时会发生跨设备错误
- 在 DDP 或模型并行场景中不可控

### 6.2 推荐依赖已有张量推断设备

**✅ 推荐：**

```python
noise = torch.randn_like(x_t)

device = x_t.device
time_tensor = torch.tensor([t], device=device)
mask = torch.zeros(batch_size, horizon, device=device, dtype=torch.bool)
```

### 6.3 dtype 也必须对齐

不能只对齐 device，不对齐 dtype。

```python
scale = torch.tensor(guidance_scale, device=x_t.device, dtype=x_t.dtype)
```

### 6.4 模块内部禁止偷偷迁移设备

函数内部不要随意调用：
- `.cuda()`
- `.cpu()`
- `.to("cuda")`

除非该函数的职责就是显式设备迁移，并且在命名与文档中写明。

## 7. 配置文件与魔法数字规范（Config & Magic Numbers）

严禁在代码中写死阈值、步数、尺度系数或实验超参数。所有可调参数必须进入统一配置系统管理。

### 7.1 必须配置化的参数

至少包括：
- 时序窗口大小 `K`
- SDF 截断距离 `sdf_truncation`
- 碰撞安全阈值 `safety_margin`
- 引导尺度 `guidance_scale`
- Warm-start 噪声比例 `warm_start_ratio`
- 采样步数、噪声调度、loss 权重
- 地图更新频率、模型同步频率、线程 sleep 间隔

### 7.2 推荐使用 dataclass / OmegaConf

**✅ 推荐：**

```python
from dataclasses import dataclass

@dataclass
class GuidanceConfig:
    safety_margin: float = 0.3
    guidance_scale: float = 10.0
    sdf_truncation: float = 0.1
    max_grad_norm: float = 10.0
```

### 7.3 禁止散落常数

**❌ 严禁：**

```python
if sdf_val < 0.3:
    loss = loss + 10.0 * penalty
```

**✅ 推荐：**

```python
if sdf_val < cfg.guidance.safety_margin:
    loss = loss + cfg.guidance.guidance_scale * penalty
```

### 7.4 配置必须可追溯

训练、评测、推理脚本必须：
- 记录实际生效配置
- 保存到实验目录
- 明确覆盖链路（默认值 / yaml / 命令行覆盖）

## 8. 日志、监控与调试规范

### 8.1 禁止使用无控制的 `print`

开发阶段可以临时使用，但提交前必须替换为统一 logger。

### 8.2 日志分级要求

- `debug`：shape、统计量、临时诊断
- `info`：关键阶段启动、配置摘要、模型加载
- `warning`：fallback、生效降级、异常值跳过
- `error/exception`：流程失败、不可恢复错误

### 8.3 关键统计建议记录

建议记录：
- GPU 显存占用
- 轨迹碰撞率
- NaN 梯度出现次数
- mapping / planning 线程频率
- 扩散采样耗时
- 模型同步耗时

## 9. PyTorch 工程实践规范

### 9.1 `optimizer.zero_grad(set_to_none=True)` 优先

这通常比将梯度置零更省内存、更高效。

```python
optimizer.zero_grad(set_to_none=True)
```

### 9.2 禁止随意原地操作

原地操作可能破坏 autograd 或造成难以定位的数值问题。

对以下场景尤其谨慎：
- 参与梯度计算的 tensor
- 被多个分支复用的 tensor
- 从缓存中取出的共享 tensor

### 9.3 控制 CPU/GPU 数据搬运

禁止在循环中频繁 `.cpu().numpy()` 做调试或日志统计，除非明确知道性能成本。

### 9.4 明确训练 / 推理模式

所有 `nn.Module` 在调用前必须根据语义设置：
- 训练：`model.train()`
- 推理：`model.eval()`

不要依赖默认状态。

## 10. 测试规范

### 10.1 核心模块必须具备最小单元测试

优先覆盖：
- shape 是否符合预期
- device / dtype 是否保持一致
- 引导步骤是否断图
- NaN fallback 是否生效
- 配置读取是否正确覆盖

### 10.2 至少包含以下测试类型

1. **Shape test**：输入多组合法 shape，检查输出 shape
2. **Device test**：CPU 与 GPU（若可用）行为一致
3. **Numerical sanity test**：输出是否有限
4. **Regression test**：关键 bug 修复后必须补测试

### 10.3 OOM 风险逻辑必须做小规模回归测试

例如 guided sampling：
- 连续多步运行
- 监控显存是否单调上涨
- 检查返回张量是否仍然带图

```python
assert output.grad_fn is None, "guided output should be detached"
```

## 11. 代码组织与命名规范

### 11.1 文件职责单一

- 一个文件只承载一个明确领域：模型、采样器、损失、数据、工具、配置
- 不要把训练循环、模型定义、调试脚本混在同一文件

### 11.2 命名要体现语义

**推荐命名：**
- `rgb_history`, `depth_history`
- `collision_penalty`, `guidance_gradient`
- `shared_isdf_model`, `inference_isdf_model`

**不推荐：**
- `x1`, `tmp2`, `new_x`, `data_final`

### 11.3 公共函数必须可复用

如果函数依赖外部全局状态，必须显式传入或封装成对象，不允许“偷偷读全局变量”。

## 12. Git 提交规范（Semantic Commits）

采用语义化提交，便于快速回溯实验变更。

### 12.1 提交格式

```text
<type>: [scope] <message>
```

### 12.2 推荐类型

- `feat`: 新功能
- `fix`: Bug 修复
- `refactor`: 重构，不改变外部行为
- `perf`: 性能优化
- `test`: 测试相关
- `docs`: 文档变更
- `tune`: 超参数或实验配置调整
- `chore`: 构建、工具链、杂项维护

### 12.3 示例

```text
feat: [Temporal] Add Transformer encoder for history frames
fix: [iSDF] Resolve OOM due to retained autograd graph in guidance step
tune: [Diffusion] Adjust noise schedule for better warm-start performance
docs: [Standards] Add tensor shape documentation requirements
```

### 12.4 提交信息要求

- 说清楚改了什么
- 尽量说明为什么改
- 不要使用 `update`, `fix bug`, `misc changes` 这类无信息量描述

## 13. 合并前检查清单（PR Checklist）

提交前必须逐项自查：

- [ ] 核心函数是否写了输入输出 shape 注释
- [ ] 复杂维度变换是否有行内说明
- [ ] 推理期求导是否严格局部开启、局部关闭
- [ ] 返回到下一时间步的 tensor 是否已经 `detach`

## 15. 附：推荐代码模板

### 15.1 安全的推理期引导模板

```python
@torch.no_grad()
def guided_step(
    diffusion_model,
    isdf_model,
    x_t: torch.Tensor,
    t: int,
    context: torch.Tensor,
    cfg,
) -> torch.Tensor:
    """
    单步扩散去噪 + SDF 物理引导。

    Args:
        x_t: 当前噪声轨迹。Shape: (B, T, D)
        context: 条件上下文。Shape: (B, D_model)

    Returns:
        x_guided: 引导后的上一时刻轨迹。Shape: (B, T, D)
    """
    pred_noise = diffusion_model(x_t, t, context)
    x_prev = remove_noise_formula(x_t, pred_noise, t)

    with torch.enable_grad():
        x_prev = x_prev.detach().requires_grad_(True)
        sdf_val = isdf_model(x_prev)
        loss = compute_collision_penalty(sdf_val, cfg.guidance.safety_margin)
        grad = torch.autograd.grad(loss, x_prev, retain_graph=False, create_graph=False)[0]

    if not torch.isfinite(grad).all():
        grad = torch.zeros_like(grad)

    grad_norm = grad.norm(dim=-1, keepdim=True).clamp_min(1e-6)
    grad = grad * (cfg.guidance.max_grad_norm / grad_norm).clamp(max=1.0)

    x_guided = x_prev - cfg.guidance.guidance_scale * grad.detach()
    return x_guided.detach()
```
