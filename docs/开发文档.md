## 2. 项目背景与系统目标

TiVP 是在原版 VIPlanner 基础上的一次系统级升级。升级重点不只是把视觉规划器换成时序扩散模型，而是构建一个“**时序先验 + 连续几何场 + 物理梯度引导**”的双轨具身导航架构，使规划系统同时具备：

- 对历史观测的时序理解能力；
- 对未知局部环境的实时几何建模能力；
- 在推理时对碰撞风险进行可微干预的能力；
- 在闭环控制中输出更平滑、更可执行、更安全的轨迹。

### 2.1 设计目标

TiVP 的目标不是单一的“轨迹生成”，而是同时满足下面四类约束：

1. **可行性**：轨迹需满足机器人平台的基础运动学常识，不出现大幅跳变、倒退振荡、瞬间急转等明显不合理行为。
2. **安全性**：轨迹在局部障碍物附近具备足够安全间隔，并可通过 SDF 梯度在采样过程中主动规避碰撞。
3. **实时性**：在线规划线程需运行在控制频率附近；在线建图线程需持续更新局部 SDF，不明显拖慢主控制回路。
4. **工程可维护性**：离线训练、在线建图、在线推理需要模块解耦，便于单独替换、单元测试与仿真联调。

### 2.2 推荐性能指标（建议实现）

原始文档未给出量化 KPI。为了便于验收，建议在开发阶段采用如下目标：

- 规划线程频率：**15–25 Hz**
- iSDF 在线更新频率：**8–15 Hz**
- 单次规划输出时域：**1.5–3.0 s**
- 轨迹预测点数 `L`：**10–40**
- 历史视觉窗口 `K`：**4–12**
- 局部地图半径：**3–10 m**
- 推理中 SDF 查询点数（单次 denoise step）：按 `B × L` 组织，优先控制在 GPU 可实时处理范围内

这些指标属于工程建议值，最终以机器人平台算力和仿真结果为准。

## 3. 系统总览

TiVP 在在线运行阶段由两个并行异步线程组成：

1. **线程 A：iSDF 在线建图线程**  
   输入 RGB-D 和位姿流，持续更新局部神经 SDF。
2. **线程 B：时序扩散规划线程**  
   输入视觉历史与上一帧轨迹，利用扩散模型生成轨迹，并在每个去噪阶段接入 iSDF 梯度进行几何修正。

这两个线程通过**共享的 iSDF 网络权重**连接：

- 线程 A 负责更新权重；
- 线程 B 读取最新权重进行 SDF 查询与梯度计算；
- 两者异步执行，不要求每个控制周期都阻塞等待对方。

### 3.1 架构核心思想

#### 3.1.1 时序扩散先验

扩散模型不是只看“当前图像”，而是接收一个历史窗口 `[t-K, ..., t]` 的视觉特征序列。这样做的目的在于：

- 捕获运动趋势与历史惯性；
- 学到更加平滑、连贯的轨迹分布；
- 避免单帧视觉噪声导致的局部不稳定规划。

#### 3.1.2 增量神经距离场

iSDF 持续用流式 RGB-D 对局部几何进行拟合。它不是离线全局地图，也不是静态 TSDF 网格，而是一个可微分的、可增量更新的连续距离场表示。其价值在于：

- 可以随环境变化实时更新；
- 可对任意轨迹点求 SDF 与梯度；
- 适合接入扩散采样过程做物理引导。

#### 3.1.3 推理期物理梯度引导

扩散模型输出的是“分布意义上的合理轨迹意图”，但真实环境中仍可能出现贴边、穿模或对局部新障碍不敏感的问题。TiVP 在每个反向去噪步骤中引入：

- 当前候选轨迹点的 SDF 值；
- 基于碰撞损失的梯度；
- 将梯度作为几何偏置注入 denoise update。

这相当于给扩散过程增加一个“可微物理裁判”，让轨迹在生成过程中逐步远离障碍区域。

## 4. 在线执行流程

### 4.1 线程 A：iSDF 在线建图流程

```text
RGB-D Frame + Robot Pose
    ↓
关键帧筛选 / 插入 Keyframe Buffer
    ↓
从关键帧中采样 rays / depth points / free-space points
    ↓
iSDF 网络前向推理 SDF(x)
    ↓
计算损失：Surface Loss + Free-space Loss + Eikonal Loss
    ↓
反向传播更新 iSDF 参数
    ↓
把最新参数保存在共享权重中
```

#### 输入

- 当前 RGB 图像 `I_t`
- 当前深度图 `D_t`
- 当前位姿 `T_world_base(t)` 或 `T_world_cam(t)`
- 相机内参 `K_cam`

#### 输出

- 已更新的共享 iSDF 权重
- 局部可查询神经 SDF 场

### 4.2 线程 B：时序扩散规划流程

```text
历史视觉窗口 [t-K:t]
    ↓
逐帧视觉 backbone 特征提取
    ↓
Temporal Encoder 聚合上下文
    ↓
上一帧轨迹 warm-start + 加噪
    ↓
反向去噪循环
    ├─ 扩散网络预测轨迹去噪方向 / 噪声
    ├─ 用共享 iSDF 计算候选轨迹的 SDF 与梯度
    ├─ 根据 collision loss 梯度修正候选轨迹
    └─ 得到下一步 denoise 结果
    ↓
输出最终局部轨迹
    ↓
发送给底层控制器
```

#### 输入

- 历史视觉观测窗口
- 当前机器人状态
- 上一个控制周期的局部轨迹
- 线程 A 输出的最新 iSDF 参数

#### 输出

- 当前控制周期的局部参考轨迹 `traj_t`

## 6. 模块详细说明

## 6.1 `viplanner/models/plannernet.py`

### 模块职责

用于提取单帧视觉特征，作为 Temporal Encoder 的输入。该模块只做“帧级编码”，不承担时序建模。

### 推荐输入输出

- 输入：`(B, C, H, W)`
- 输出：`(B, F)`

其中：

- `B`：批大小
- `C`：通道数（RGB 时通常为 3）
- `H, W`：图像尺寸
- `F`：视觉特征维度

### 推荐实现

可选实现路线：

1. 轻量 CNN Backbone（如 ResNet18/34 截断版）
2. MobileNet / EfficientNet 类轻量网络
3. 若已有原版 VIPlanner backbone，则优先复用其结构与权重接口

### 建议类定义

```python
class PlannerNet(nn.Module):
    def __init__(self, in_channels: int = 3, feat_dim: int = 256):
        ...

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """x: (B, C, H, W) -> feat: (B, F)"""
```

### 开发要求

- 输出必须是固定维度向量，便于组装时序序列。
- 若 backbone 输出为 feature map，需在模块内完成 pooling / flatten。
- 训练时允许冻结 backbone 前几层，以稳定时序扩散训练。

## 6.2 `viplanner/models/temporal_encoder.py`

### 模块职责

对过去 `K` 帧视觉特征进行时序建模，输出一个聚合后的上下文向量 `Context Vector`，供条件扩散网络使用。

### 推荐输入输出

- 输入：`(B, K, F)`
- 输出：`(B, D_ctx)`

其中：

- `K`：时间窗口长度
- `F`：单帧特征维度
- `D_ctx`：时序上下文维度

### 推荐实现结构

1. 输入投影层：把 `F` 投到 Transformer 所需维度 `D_model`
2. 位置编码：使用 sinusoidal positional encoding 或 learnable positional embedding
3. Transformer Encoder：1~4 层
4. 聚合头：
   - 可取最后一个 token；
   - 可做平均池化；
   - 推荐增加 `[CLS]` token，统一输出聚合向量

### 建议类定义

```python
class TemporalEncoder(nn.Module):
    def __init__(
        self,
        input_dim: int,
        model_dim: int,
        num_heads: int,
        num_layers: int,
        ff_dim: int,
        dropout: float,
        max_seq_len: int,
        out_dim: int,
    ):
        ...

    def forward(self, feats: torch.Tensor) -> torch.Tensor:
        """feats: (B, K, F) -> context: (B, D_ctx)"""
```

### 开发细节

- 必须明确位置编码是否计入 `[CLS]` token。
- 历史窗口中若存在缺帧，需要在 dataset 或 collate_fn 中补齐，Temporal Encoder 本身不建议处理时长不齐问题。
- 为了在线推理稳定性，推荐让 `K` 固定。

### 测试点

- 输入不同 `K` 时，位置编码是否越界。
- batch 首维变动时是否广播正确。
- 输出是否始终为 `(B, D_ctx)`。

## 6.3 `viplanner/models/diffusion_unet.py`

### 模块职责

该模块是轨迹扩散模型的核心。它对加噪后的轨迹进行条件去噪，输出预测噪声、预测残差或预测净轨迹（依据 diffusion parameterization）。

### 推荐输入输出

- 轨迹输入：`x_t`，形状 `(B, C, L)`
- 扩散时间步：`t`，形状 `(B,)` 或标量
- 上下文条件：`context`，形状 `(B, D_ctx)`
- 输出：`eps_pred` 或 `x0_pred`，形状 `(B, C, L)`

其中：

- `C = 2` 或 `3`，表示局部轨迹点维度
- `L`：轨迹预测长度

### 推荐结构

1. 时间步嵌入（Sinusoidal + MLP）
2. 轨迹输入卷积 / 线性投影
3. 1D U-Net 编码器与解码器
4. 在每个 block 中引入条件：
   - **FiLM**：对 feature 做 scale-shift 调制；或
   - **Cross-Attention**：把 context 当作条件 token
5. 输出层映射回 `(B, C, L)`

### 建议类定义

```python
class DiffusionUNet1D(nn.Module):
    def __init__(
        self,
        traj_dim: int,
        context_dim: int,
        base_channels: int,
        channel_mults: list[int],
        num_res_blocks: int,
        time_embed_dim: int,
        cond_mode: str = "film",
    ):
        ...

    def forward(
        self,
        x_t: torch.Tensor,
        timesteps: torch.Tensor,
        context: torch.Tensor,
    ) -> torch.Tensor:
        """(B, C, L) -> (B, C, L)"""
```

### 参数化建议

建议优先支持三种 parameterization，便于后续试验：

- `predict_epsilon`
- `predict_x0`
- `v_prediction`

默认建议从 `predict_epsilon` 开始实现，便于与标准 DDPM/DDIM 兼容。

### 工程注意事项

- 轨迹张量在数据集里常为 `(B, L, C)`，进入 1D 卷积前必须转为 `(B, C, L)`。
- 输出给控制器时通常又需要转回 `(B, L, C)`。
- FiLM 条件注入实现成本较低，建议 Phase 1 先采用 FiLM，Cross-Attention 作为可选升级。

## 6.4 `viplanner/models/isdf_network.py`

### 模块职责

输入任意 3D 点坐标，输出其 Signed Distance 值，用于局部障碍场建模、碰撞检测与轨迹引导。

### 推荐输入输出

- 输入：`(B, N, 3)` 或 `(N, 3)`
- 输出：`(B, N, 1)` 或 `(N, 1)`

### 推荐结构

1. 坐标编码：
   - Fourier Positional Encoding；或
   - Hash Grid Encoding（若引入 tiny-cuda-nn 等加速依赖）
2. 轻量 MLP：4~8 层
3. 输出单标量 SDF

### 建议类定义

```python
class ISDFNetwork(nn.Module):
    def __init__(
        self,
        in_dim: int = 3,
        hidden_dim: int = 128,
        num_layers: int = 6,
        encoding_type: str = "fourier",
        sdf_scale: float = 1.0,
    ):
        ...

    def forward(self, points: torch.Tensor) -> torch.Tensor:
        """points: (..., 3) -> sdf: (..., 1)"""
```

### 损失设计建议

建议至少包含以下损失：

1. **Surface Loss**：表面点 SDF 应接近 0
2. **Free-space Loss**：可见自由空间点 SDF 应大于 0（或满足相应符号约定）
3. **Truncation / Near-surface Loss**：增强表面附近拟合稳定性
4. **Eikonal Loss**：约束 `||∇SDF(x)|| ≈ 1`

### 符号约定要求

必须在项目内统一以下规则并写入代码注释：

- 障碍物内部是负值还是正值；
- 自由空间是正值还是负值；
- 碰撞损失 `max(0, margin - SDF(x))` 建立在何种符号定义上。

**推荐统一为：**

- 自由空间：`SDF > 0`
- 表面：`SDF = 0`
- 障碍物内部：`SDF < 0`

这样碰撞损失定义最自然。

## 6.5 `viplanner/memory/temporal_dataset.py`

### 模块职责

为离线扩散训练提供滑动窗口样本，把离散的数据帧组织成时序块，并对齐：

- 历史图像窗口 `[t-K+1, ..., t]`
- 对应目标轨迹 `traj_t`
- 可选辅助信息（速度、目标点、语义标签、碰撞标签等）

### 推荐输出字段

每个样本建议返回字典：

```python
{
    "images": Tensor[K, C, H, W],
    "traj": Tensor[L, C_traj],
    "pose": Tensor[...],
    "goal": Tensor[...],
    "valid_mask": Tensor[K],
    "meta": dict,
}
```

### 滑动窗口规则

给定连续序列索引 `0 ... T-1`：

- 历史窗口长度为 `K`
- 预测轨迹长度为 `L`
- 则合法中心时刻 `t` 至少满足：
  - 有足够历史：`t >= K - 1`
  - 有足够未来轨迹标注：`t + L - 1 < T`

### 推荐类定义

```python
class TemporalTrajectoryDataset(Dataset):
    def __init__(self, root: str, hist_len: int, pred_len: int, ...):
        ...

    def __len__(self) -> int:
        ...

    def __getitem__(self, idx: int) -> dict:
        ...
```

### 关键开发要求

- 必须在 dataset 中完成**局部坐标系转换**，保证输出轨迹与视觉输入对齐到当前机器人坐标系。
- 若某些样本跨 episode 边界，必须丢弃，不能把两个轨迹段错误拼接成一个窗口。
- 必须支持训练 / 验证集切分。

### 单元测试建议

- 窗口跨序列边界时是否被过滤；
- 历史窗口和目标轨迹长度是否正确；
- 坐标变换后第一轨迹点是否与当前时刻 base frame 一致。

## 6.6 `viplanner/memory/keyframe_buffer.py`

### 模块职责

维护 iSDF 在线训练所需的关键帧缓存，使在线建图线程无需保留所有历史帧，而是以有限内存维护最有价值的视角样本。

### 核心职责

1. 管理固定长度环形队列；
2. 根据位姿变化决定是否插入关键帧；
3. 提供从缓存中采样射线和深度点的接口；
4. 为 iSDF 在线 SGD 提供训练 batch。

### 建议关键帧数据结构

```python
@dataclass
class Keyframe:
    rgb: torch.Tensor          # [C, H, W]
    depth: torch.Tensor        # [1, H, W] or [H, W]
    pose_world_cam: torch.Tensor  # [4, 4]
    intrinsics: torch.Tensor   # [3, 3]
    timestamp: float
```

### 建议类定义

```python
class KeyframeBuffer:
    def __init__(
        self,
        capacity: int,
        trans_thresh: float,
        rot_thresh_deg: float,
        device: str = "cpu",
    ):
        ...

    def should_add_keyframe(self, pose_world_cam: torch.Tensor) -> bool:
        ...

    def add(self, keyframe: Keyframe) -> None:
        ...

    def sample_rays(self, batch_size: int) -> dict:
        ...

    def __len__(self) -> int:
        ...
```

### 关键帧插入策略（建议实现）

满足任一条件可插入：

- 平移超过阈值 `trans_thresh`
- 旋转超过阈值 `rot_thresh_deg`
- 缓存为空
- 距离上次插入时间超过最大时间阈值 `dt_max`

### `sample_rays(batch_size)` 建议返回

```python
{
    "sampled_points": Tensor[N, 3],
    "surface_points": Tensor[Ns, 3],
    "free_points": Tensor[Nf, 3],
    "ray_dirs": Tensor[N, 3],
    "depth_gt": Tensor[N],
    "frame_ids": Tensor[N],
}
```

### 工程注意事项

- 在线训练时不建议每次只采样最新帧，应在最近帧和历史关键帧之间做混合采样。
- 为减少偏置，可对近距离表面点与自由空间点分层采样。
- 环形队列覆盖旧帧时，需保证线程安全。

### 模块职责

作为独立线程或独立进程运行，负责持续更新 iSDF，并向规划线程暴露最新可查询模型参数。

### 功能清单

1. 接收传感器数据并更新关键帧缓存；
2. 周期性从缓存采样训练 batch；

```python
class ISDFOnlineWorker:

        ...

        ...

    def get_shared_state(self):
        ...
```

### 多进程共享建议

若使用 PyTorch 多进程：

- 在主进程创建 `isdf_model.share_memory_()`
- 在线 worker 进程使用该共享模型做参数更新
- 规划线程只读当前模型参数
- 对于优化器状态，不建议跨线程频繁访问；必要时将参数复制到只读快照模型

### 锁机制建议

原始文档强调“共享权重”，但未规定同步方式。推荐两种模式：

#### 模式 A：无锁快照读取（推荐先实现）

- Worker 每完成若干步 SGD 后，复制参数到一份 `read_only_model`
- Planner 始终读取快照模型
- 优点：简单稳定，避免 planner 读取到中间梯度状态

#### 模式 B：读写锁

- 更新参数时加写锁
- 规划线程查询 SDF 时加读锁
- 优点：参数更新及时；缺点：控制线程可能被阻塞

Phase 3 之前优先推荐模式 A。

## 8. 推理与引导模块设计

## 8.1 `viplanner/inference/guided_sampler.py`

### 模块职责

封装完整的 DDPM/DDIM 采样流程，并在每个采样步骤中接入 iSDF 引导。

### 推荐接口

```python
class GuidedSampler:
    def __init__(self, diffusion_model, scheduler, guidance_fn=None):
        ...

    def sample(
        self,
        context: torch.Tensor,
        init_traj: torch.Tensor | None = None,
        num_steps: int | None = None,
        guidance_kwargs: dict | None = None,
    ) -> torch.Tensor:
        ...
```

### 采样核心逻辑

对于每个 denoise step `t -> t-1`：

1. 扩散网络预测 `eps_pred`
2. 根据调度器求出原始 `x_{t-1}`
3. 调用 `guidance_fn(x_{t-1})`
4. 用得到的几何修正量更新 `x_{t-1}`
5. 进入下一步

### 推荐伪代码

```python
x_t = init_noise_or_warmstart(...)
for t in reversed(timesteps):
    eps = model(x_t, t, context)
    x_prev = scheduler.step(eps, t, x_t)

    if guidance_fn is not None:
        delta = guidance_fn(x_prev, t)
        x_prev = x_prev + delta

    x_t = x_prev
return x_t
```

### 工程注意事项

- `guidance_fn` 必须可开关，便于对比有无几何引导的效果。
- 需要记录每一步 guidance 的平均修正幅度，以便调参。
- 若 guidance 过强，可能导致轨迹畸变或采样发散，需要加入裁剪或归一化。

## 8.2 `viplanner/inference/isdf_guidance.py`

### 模块职责

对候选轨迹计算碰撞损失，并基于 iSDF 的梯度生成几何修正项。

### 理论接口

给定局部轨迹点 `x ∈ R^{L×d}`，将其嵌入 3D 点后查询：

- `sdf_i = SDF(p_i)`
- 碰撞损失：

```text
L_collision = Σ_i max(0, margin - sdf_i)
```

其中 `margin > 0` 表示希望轨迹距障碍物至少保留一段安全距离。

### 建议类定义

```python
class ISDFGuidance:
    def __init__(self, isdf_model, margin: float, scale: float, clip_norm: float | None = None):
        ...

    def compute_loss(self, traj_local: torch.Tensor) -> torch.Tensor:
        ...

    def compute_delta(self, traj_local: torch.Tensor, t: int | torch.Tensor) -> torch.Tensor:
        ...
```

### 轨迹到 SDF 查询点的映射

如果 diffusion 输出的是二维平面轨迹 `(x, y)`，则必须明确其对应到 3D 点的方式。建议统一：

- 地面机器人：`p = [x, y, z_ref]`
- 其中 `z_ref` 可固定为机器人底盘中心高度或规划平面高度

若机器人机体有体积，建议扩展为多点碰撞近似：

- 中心点
- 左右边界点
- 前后边界点

这样 SDF guidance 更接近真实碰撞体积。

### Guidance 输出建议

推荐输出：

```text
delta = λ(t) * normalize(-∇_x L_collision)
```

或等价形式：

```text
x_{t-1} = x_{t-1} - λ(t) * ∇_x L_collision
```

其中：

- `λ(t)`：随时间步变化的 guidance 系数
- 早期 denoise step 可弱一些，后期可强一些，或反之，需要实验决定

### 稳定性建议

- 对梯度做 `clip_norm`
- 对过小 SDF 的点采用更强惩罚
- 对已远离障碍的点不施加修正
- 若所有点都安全，则直接返回零修正

## 8.3 `viplanner/inference/warm_start.py`

### 模块职责

利用上一控制周期的轨迹作为当前周期的初始化，减少扩散采样步数，提高在线规划频率与轨迹连续性。

### 核心思路

给定上一周期轨迹 `traj_{t-1}`：

1. 按当前控制前进量截断已执行部分；
2. 把剩余部分左移对齐；
3. 末尾补齐若干点（复制末点或线性外推）；
4. 将该轨迹加噪到较低噪声时间步 `T_start < T`；
5. 从 `T_start` 开始执行反向采样，而非从纯高斯噪声开始。

### 建议类定义

```python
class WarmStarter:
    def __init__(self, total_steps: int, start_step: int):
        ...

    def prepare(self, prev_traj: torch.Tensor, executed_len: int = 1) -> torch.Tensor:
        ...

    def diffuse_to_start_step(self, traj: torch.Tensor, noise: torch.Tensor) -> torch.Tensor:
        ...
```

### 开发要点

- 必须处理第一帧没有上一轨迹的情况，此时退化为随机初始化。
- 必须处理上一轨迹与当前局部坐标系不一致的问题。建议先把上一轨迹变换到当前 base frame 再 warm-start。
- 推荐记录 warm-start 命中率、起始步数与采样耗时。

## 9. 关键数据流与张量规范

本节是整个项目最关键的工程约束之一。很多问题不是模型能力不足，而是数据维度和坐标系不一致导致。

## 9.1 轨迹张量格式

项目内部建议统一两种表示：

### 表示 A：逻辑表示（便于几何处理）

- `(B, L, C)`

含义：

- `B`：batch size
- `L`：轨迹长度
- `C`：2 或 3

适用于：

- 数据集输出
- 坐标变换
- 控制器接口
- 可视化

### 表示 B：网络表示（便于 1D Conv）

- `(B, C, L)`

适用于：

- Diffusion UNet 前向
- 1D 卷积 block

### 强制要求

- 任何张量在进入 `diffusion_unet` 前，必须明确调用 `.transpose(1, 2)` 或等价操作；
- 任何张量在返回控制器前，必须回到 `(B, L, C)`。

推荐在工具函数中统一封装：

```python
def traj_blc_to_bcl(x): ...
def traj_bcl_to_blc(x): ...
```

---

## 9.2 坐标系规范

### 强制要求

送入 Diffusion 模型和送入 SDF 评估的轨迹点，必须全部转换到**机器人当前时刻局部坐标系（Local Base Link）**。

### 推荐原因

1. 扩散模型更容易学习局部运动模式；
2. SDF 局部场查询与局部规划天然一致；
3. Warm-start 跨周期对齐更方便；
4. 不依赖全局地图尺度与漂移。

### 推荐坐标变换链

- 全局位姿：`T_world_base(t)`
- 若轨迹点原始在 world frame，转换为 local frame：

```text
p_local = T_base_world(t) * p_world
```

### 工程建议

在 `utils/transforms.py` 中统一实现并单测：

- `world_to_local_points(...)`
- `local_to_world_points(...)`
- `transform_traj(...)`

### 单位约束

- 统一使用**米（meters）**
- 深度图若原始单位为毫米，必须在最前面完成转换

---

## 9.3 视觉历史窗口格式

推荐数据流：

1. dataset 输出：`(B, K, C, H, W)`
2. reshape 为 `(B*K, C, H, W)` 送入 `PlannerNet`
3. 输出 `(B*K, F)`
4. reshape 回 `(B, K, F)`
5. 送入 `TemporalEncoder`

推荐封装成辅助函数，避免在 trainer 和 inference 中重复手写维度变换。

---

## 9.4 iSDF 查询格式

推荐统一：

- 单帧轨迹：`(L, 2)` 或 `(L, 3)`
- 批量轨迹：`(B, L, 2)` 或 `(B, L, 3)`
- 查询前展平为：`(B*L, 3)`
- SDF 输出再 reshape 回：`(B, L, 1)`

若采用机器人体积采样，则维度变为：

- `(B, L, N_body, 3)` -> flatten 为 `(B*L*N_body, 3)`

## 10. 训练流程详细设计

## 10.1 Phase 1：时序 Diffusion 离线训练

### 目标

在离线数据集上训练一个能够根据历史视觉上下文生成无碰撞、平滑局部轨迹的扩散先验。

### 输入

- 历史 RGB 序列
- 对应局部轨迹标签

### 输出

- `PlannerNet` 权重
- `TemporalEncoder` 权重
- `DiffusionUNet1D` 权重

### 推荐训练步骤

1. 加载样本：`images[K] + traj[L]`
2. 提取每帧视觉特征
3. 聚合得到 `context`
4. 对 GT 轨迹采样随机扩散步 `t`
5. 加噪得到 `x_t`
6. 模型预测噪声 `eps_pred`
7. 计算 loss，反向传播

### 推荐配置项

```yaml
model:
  feat_dim: 256
  ctx_dim: 256
  traj_dim: 2
  pred_len: 20

diffusion:
  num_train_steps: 1000
  beta_schedule: cosine
  prediction_type: epsilon

train:
  batch_size: 32
  lr: 1e-4
  epochs: 100
  grad_clip: 1.0
```

### 验收标准（建议）

- 训练 loss 稳定下降
- 验证集 ADE / FDE 收敛
- 轨迹平滑度优于无时序版本
- 在静态障碍离线回放中，碰撞率明显低于基线

---

## 10.2 Phase 2：iSDF 单独重建

### 目标

在静态或半静态数据上，先验证 iSDF 网络、关键帧缓存和在线训练逻辑是否正确。

### 推荐步骤

1. 从录制数据中依次读取 RGB-D + Pose
2. 插入关键帧缓存
3. 周期性采样 rays / points
4. 优化 iSDF 网络
5. 通过切片或 query grid 验证距离场形状

### 推荐可视化

- 2D 水平切片热力图
- 若干固定 query 点的 SDF 时序曲线
- 表面点云与零等值面近似可视化

### 验收标准（建议）

- 表面附近 SDF 逼近 0
- 空闲区域 SDF 正值稳定
- Eikonal loss 逐步收敛
- 随帧增量更新后地图不会快速崩坏

## 10.3 Phase 3：静态引导融合验证

### 目标

在不引入多线程复杂度前，先验证“扩散轨迹 + iSDF 梯度干预”这条核心链路能否成立。

### 推荐实验

1. 取一段静态场景数据；
2. 用离线训练好的 diffusion 输出轨迹；
3. 对一部分故意穿模的轨迹点施加 guidance；
4. 比较 guidance 前后的轨迹碰撞率与平滑度。

### 关键观测指标

- guidance 前后最小 SDF 值
- 轨迹平均偏移量
- 终点偏差
- 采样时间增加量

### 验收标准（建议）

- 能显著提升穿障轨迹的最小安全距离
- 不出现大范围振荡或明显路径畸变
- 推理耗时增长可控

## 10.4 Phase 4：闭环动态仿真测试

### 目标

在仿真平台（如 Isaac Sim）中进行完整闭环验证，测试动态环境和异步线程系统行为。

### 需联通模块

- 传感器接口
- KeyframeBuffer
- iSDF worker
- GuidedSampler
- WarmStarter
- Controller

### 重点关注问题

- 线程同步延迟
- 地图陈旧导致的 guidance 失效
- Warm-start 在快速转向时失真
- 控制器与规划轨迹时基不一致

### 验收标准（建议）

- 系统可持续运行，无明显内存泄漏
- 机器人可在动态障碍环境中完成导航任务
- 轨迹刷新平滑，不发生频繁重规划抖动
- 相比无 guidance 版本，碰撞率下降

## 11. 推理时序与线程协同设计

## 11.1 推荐主循环

### Mapping Worker

```python
while running:
    sensor_packet = sensor_queue.get()
    keyframe_buffer.update(sensor_packet)

    if len(keyframe_buffer) >= min_frames:
        batch = keyframe_buffer.sample_rays(batch_size)
        loss = train_isdf_one_step(batch)

```

### Planner Loop

```python
while control_running:
    obs_hist = obs_buffer.get_last_k()
    context = encode_history(obs_hist)

    init_traj = warm_starter.prepare(prev_traj) if prev_traj is not None else None
    traj = guided_sampler.sample(context, init_traj, guidance_kwargs={...})

    controller.send(traj)
    prev_traj = traj
```

## 11.2 时间一致性建议

由于线程 A 和线程 B 异步运行，规划线程读到的 iSDF 未必对应同一时刻传感器。为降低时序错位影响，建议：

- 给每次发布的 iSDF 快照附带时间戳；
- 规划日志记录当前传感器时间与地图快照时间差；
- 若时间差超过阈值，可降低 guidance 强度或直接跳过本轮 guidance。

## 14. 错误处理与回退策略

在线系统一定要设计回退逻辑，避免单个模块失败导致整机不可用。

### 14.1 iSDF 不可用时

若 iSDF worker 尚未热启动完成，或当前快照不可读：

- 允许 planner 退化为无 guidance 的纯 diffusion 采样；
- 日志中标记当前为 `guidance_disabled`。

### 14.2 Warm-start 不可用时

若上一轨迹不存在或坐标变换失败：

- 退化为纯随机初始化；
- 不应中断主控制循环。

### 14.3 Guidance 过强时

若检测到单步修正过大、轨迹发散或 NaN：

- 对修正量裁剪；
- 或跳过本步 guidance；
- 必须在日志中保留错误状态与统计计数。

### 14.4 传感器时间戳异常时

- 若 RGB、Depth、Pose 时差过大，则该帧不入关键帧缓存；
- 对外发出 warning，但不阻塞主线程。

## 16. 性能优化建议

## 16.1 计算优化

- PlannerNet 可做半精度推理
- Diffusion 采样优先尝试 DDIM 以减少步数
- iSDF 查询点尽量批量化，不要 Python for-loop 单点查询
- 对历史图像特征可做短时缓存，避免重复编码

## 16.2 内存优化

- KeyframeBuffer 仅保存必要分辨率的 RGB-D
- 在线 worker 只保留固定数量关键帧
- 若 GPU 紧张，可把关键帧缓存留在 CPU，采样后再搬运 mini-batch

## 16.3 实时性优化

- Warm-start 减少采样步数
- guidance 不必每一步都启用，可隔步启用
- 对安全区域轨迹点跳过 SDF 梯度计算

## 19. 推荐开发顺序

为了降低研发风险，建议按如下顺序推进，而不是并行大范围开工：

1. **先打通数据和 shape 规范**  
   先完成 `temporal_dataset.py`、轨迹坐标转换、统一 tensor format。

2. **再完成时序扩散主干**  
   先验证不接 iSDF 时，扩散模型能稳定生成轨迹。

3. **单独验证 iSDF 重建**  
   确保 SDF 值和梯度方向可信，再做引导。

4. **静态场景融合 guidance**  
   只在离线回放里测 guidance，避免把多线程问题和算法问题混在一起。

5. **最后再上异步系统和闭环仿真**  
   这样定位问题更清晰，调试成本最低。

## 20. 风险点与规避建议

### 风险 1：轨迹坐标系不统一

**后果**：扩散模型学到错误分布，guidance 方向失效。  
**规避**：所有数据入口和推理入口都打印/检查 local frame 对齐结果。

### 风险 2：SDF 符号约定混乱

**后果**：引导方向反了，轨迹越推越撞。  
**规避**：在 `isdf_network.py` 和 `isdf_guidance.py` 顶部写清楚符号定义，并做单元测试。

### 风险 3：guidance 过强导致采样崩坏

**后果**：轨迹畸形、振荡、终点偏移严重。  
**规避**：引入 scale schedule、clip norm 与安全阈值。

### 风险 4：多线程共享模型状态不稳定

**后果**：planner 读取到中间态参数，出现偶发 NaN。  
**规避**：先使用只读快照模型，再考虑更复杂共享方案。

### 风险 5：在线系统算力不足

**后果**：控制频率掉帧。  
**规避**：优先减少 diffusion 步数、减少 iSDF query 数、启用 warm-start。

## 21. 结论

TiVP 的难点不在某一个网络，而在于三部分能否形成工程闭环：

1. **Temporal Encoder + Diffusion UNet** 提供符合时序语义的轨迹先验；
2. **iSDF Online Worker** 提供可微、实时更新的局部几何约束；
3. **Guided Sampler + Warm Start** 把先验、几何与实时控制整合为可运行的在线系统。

因此，开发策略必须坚持“**先单模块正确，再两两融合，最后闭环联调**”的原则。只要严格执行本文档中的形状规范、坐标系规范、模块边界和阶段验收标准，TiVP 就可以从概念方案逐步落地为一个可训练、可部署、可扩展的具身导航系统。
